# Project Overview

## 核心困境

### 物理层限制

BLE设计初衷是个人局域网，其链路预算通常设计为覆盖10米范围。但是，在复杂的建筑环境，或者异地控制场景下，物理信号遵循反平方定律，发射功率会在极有限的传输距离内衰减到不可用。我们需要一个虫洞，将RF信号转换为IP数据包，上传云端传输，来穿越物理障碍。



## 技术路线决策

### 路线一：逆向工程与协议破解（已否决）

“为什么不直接破解蓝牙协议？”，如果可以实现，这样的方案无疑是最方便的。我们甚至不需要接收到真实的蓝牙信号，只需要完全理解广播设备生成信号的原始数据和加密方式，就可以复刻出完全相同的信号。问题在于，这种方案非常脆弱。

#### 1. 链路层加密

现代BLE设备普遍采用了AES-CCM模式。这意味着数据不仅被AES-128加密，还包含完整性校验。更致命的是Rolling Codes，即使我们抓到了同一个操作的两个数据包，它们的payload也完全不同，单纯的重放攻击在很多设备上是无效的。

#### 2. 隐私特性

RPA会使得设备周期性更换MAC地址，如果不掌握厂商的Identity Resolving Key，我们甚至无法持续追踪同一个设备。

#### 3. 维护成本

即使我们真的逆向解析出了厂商的IRK，逆向工程还需要不断与厂商的更新赛跑，每当厂商OTA升级了固件，修改了payload结构或者加密种子，网关就会失效。



### 路线二：透明隧道转发（采用方案）

我们不尝试理解Payload中的密文含义，我们只负责将raw data忠实地传输到云端，再通过接收端抓取并广播。这样，不论厂商使用了多复杂的加密，多频繁更新固件，都与我们的设计无关。我们只搭建了一个信号中继节点，而这已经可以满足我们的需求。



## 载体选型

### 方案一：Android / iOS网关（已否决）

手机自带蓝牙和WiFi，为什么不拿一个旧手机作为网关呢？在工程实践中，手机有缺陷。

#### 1. 操作系统调度机制

现代操作系统极其激进。为了省电，OS 会强制挂起后台进程，甚至回收 Socket 连接。虽然可以申请Wake Lock或前台服务，但这极其不稳定，且受限于厂商定制的杀后台策略。这导致数据传输的不可预测延迟。

#### 2. 硬件控制权缺失

手机无法精细控制蓝牙芯片的底层行为，如调整扫描窗口和扫描间隔。无法外接高增益天线，只能依赖手机内置的天线，可能导致信号接收不稳定。

**但是说白了，这些问题真的是问题吗？并非不可逾越。实测目标设备在数分钟内不改变发布数据包，延迟在该设备下不要紧。此外，既然设备本身就是为手机接收配置的，那手机就必然可以在一定范围内收到信号，信号强度也不是问题。那为什么还是放弃了这个方案呢？因为用手机，要在臃肿的操作系统上编写应用层代码，还要破解系统获取控制权，这并非我最感兴趣的事。**



### 方案二：ESP32微控制器（采用方案）

#### 1. 实时性

运行 FreeRTOS，我们可以精确控制每一个 CPU 时钟周期。没有“后台杀进程”的概念，只要有电，任务就在跑。

#### 2. 射频共存机制

ESP32 硬件层面集成了Bluetooth / WiFi Coexistence Arbiter。通过分时复用（Time Division Multiplexing），它能极其高效地处理一边收蓝牙，一边发 WiFi的场景，这是手机上层应用难以触及的优化领域。

#### 3. 极致的电源管理

我们可以进入深度睡眠，功耗低至10µA，仅保留RTC内存和GPIO唤醒逻辑。配合电源管理芯片，可以实现真正的电池供电，超长待机。同时加入了PD诱骗方案，实现快充。



## 系统拓扑

#### 1. 接收端：

* 基于 ESP32 + PMIC。    
* 负责 7x24 小时待机，抓取空中的 Raw Data。
* 通过 MQTT 将数据传输至云端。 

#### 2. 广播端：

* 基于 ESP32。    
* 订阅 MQTT Topic，收到数据后在本地复刻广播。

#### 3. 云端（MQTT）：

- 作为数据中转站，解耦了物理空间的限制。
